# Development Plan

## 目标摘要

当前项目已经从“能收数据、能简单驱动”的原型阶段，进入“固化中间层、稳定调试链路、为正式目标适配做准备”的阶段。

当前总目标没有变化：

1. 固化实时接收链路
2. 固化中间层预览对象
3. 把预览层和目标层彻底解耦
4. 让映射与预设成为正式配置能力
5. 以 `Auto Rig Pro` 骨架和 `MMD` 面部为下一阶段主要适配目标

## 当前开发状态

### 已完成

#### 1. 模块化拆分

项目已完成首轮模块化，主逻辑不再集中在单一文件中。

当前主要模块包括：

- `constants.py`
- `properties.py`
- `network.py`
- `runtime.py`
- `mapping.py`
- `presets.py`
- `dummy_vrm.py`
- `ui_main.py`
- `ui_preview.py`
- `ui_mapping.py`
- `ui_intermediate.py`

#### 2. UI 中文化

所有主要用户可见 UI 已改为中文，包括：

- 主面板
- 中间层面板
- 映射面板
- 数据预览面板
- operator 按钮文本
- 运行状态和错误提示

#### 3. 中间层对象固化

项目当前已明确区分两套 Scene 槽位：

- 目标层：
  - `vmc_link_armature`
  - `vmc_link_face_object`
- 预览层：
  - `vmc_link_preview_armature`
  - `vmc_link_preview_face_object`

兼容老场景的自动迁移逻辑也已完成。

#### 4. 预览层与目标层解耦

当前运行时已经拆成两条驱动链：

- 预览层：
  - 预览骨架按原始 `VMC` 数据驱动
  - 预览面部按原始 `ARKit 52 Key` 数据驱动
  - 不读取用户映射设置
- 目标层：
  - 继续按手动映射、自动匹配和缓存映射驱动
  - 受 `实时驱动目标对象` 控制

这一步已经完成当前阶段最重要的调试基础建设。

#### 5. 固定预览对象与集合管理

当前插件已经建立专用集合管理策略：

- 插件专用集合：`VMC_Link`
- ARKit 资产源集合：`ArkitFace`

当前已落地：

- `创建/重建 VRM 预览骨架`
  - 只在 `VMC_Link` 集合内创建和重建
  - 只处理 `VMC_TEST_DUMMY` 预览骨架
- `导入 ARKit 调试面部`
  - 只从 `assets/Arkitface.blend` 追加 `ArkitFace` 集合
  - 不再导入其他无关对象

#### 6. 数据预览重构

数据预览已从主工作流中降为二级调试功能。

当前结构：

- 位于面板最底部
- 默认折叠
- `VMC` 和 `ARKit` 数据并排显示

#### 7. 基础性能优化

当前已经完成一轮明显影响视口流畅度的优化：

- 去掉每 tick 整块 `VIEW_3D` 强制重绘
- 改成只刷新 UI 区域
- UI 刷新节流
- 预览链路增加独立缓存，减少重复名字匹配

### 当前仍未完成

- `VRM 中间层 -> Auto Rig Pro` 专用映射工作流
- `ARKit 52 Key -> MMD Shape Key` 专用映射工作流
- built-in 目标骨架 / 面部预设
- 映射完整性检查与诊断报告
- 面向实际生产适配的更强调试工具

说明：

- `VMC Blend -> 形态键` 映射能力会继续保留
- 但后续面部开发主线不再围绕 `VMC -> MMD` 展开
- 当前主要方向收敛为 `ARKit -> MMD`

## 当前架构结论

### 已固定的输入中间层

- 身体中间层：
  - `VRM` 风格标准预览骨架
- 面部中间层：
  - `ARKit 52 Key`

### 当前正式调试路径

1. 验证是否收到原始数据
2. 验证预览层是否正确实时驱动
3. 验证目标层映射是否正确
4. 验证是否能稳定保存和复用映射预设

这比早期“直接把原始数据写进目标骨架 / 目标面部”更适合持续开发。

## 后续开发路线

## Phase 1：基础链路收敛

状态：`已完成`

已完成内容：

- 模块化拆分
- 中文化 UI
- 三套映射和 JSON 预设
- 中间层骨架与面部调试对象
- 预览层 / 目标层解耦
- 集合管理
- 数据预览重构
- 一轮性能优化

## Phase 2：目标适配收敛

状态：`下一优先级`

目标：

- 把当前“通用映射能力”收敛为“对明确目标可用”

建议顺序：

1. `VRM 中间层 -> Auto Rig Pro` 骨架适配
2. `ARKit 52 Key -> MMD Shape Key` 面部适配
3. 视需要补充 `VMC Blend -> 形态键` 兼容预设

需要落地的能力：

- built-in 骨架映射预设
- built-in `ARKit -> MMD` 面部映射预设
- 缺失骨骼 / 缺失 Shape Key 提示
- 当前映射完整性检查

## Phase 3：调试工具深化

状态：`待开始`

建议补强：

- 映射完整性报告
- 当前预设来源提示
- 当前预览层对象状态汇总
- 更紧凑的数据预览摘要
- 面向目标模型的快速验证按钮

## Phase 4：进一步性能与稳定性

状态：`待开始`

当前已经完成的是“先去掉最明显的主线程浪费”。

后续需要继续观察和收敛的是：

- 大量骨骼目标的写入开销
- 大量 Shape Key 的写入开销
- 数据预览面板展开时的 UI 文本开销
- 更细粒度的脏标记和更新节流

注意：

Blender 对对象、骨骼和 Shape Key 的修改本身仍然必须在主线程进行，所以后续优化的重点应继续放在：

- 减少不必要的更新
- 降低重复匹配和重复绘制
- 缩小每次 tick 的实际工作量

而不是盲目引入线程化写入。

## 当前建议的近期执行顺序

1. 先做 `Auto Rig Pro` 骨架映射预设和验证流程
2. 再做 `ARKit -> MMD` 面部映射预设
3. 然后补完整性检查与诊断 UI
4. 最后继续做更细粒度的性能优化

## 当前验收标准

当前阶段已经可以认为基础目标达成，当下的验收重点应转为“目标适配是否可稳定落地”。

下一阶段建议以以下结果作为验收标准：

- `VRM 中间层 -> Auto Rig Pro` 映射可配置、可保存、可复用
- `ARKit 52 Key -> MMD Shape Key` 映射可配置、可保存、可复用
- 缺失映射可以快速诊断
- 预览层与目标层问题能被快速分离定位

补充：

- `VMC Blend -> 形态键` 作为备选链路保持可用即可
- 不作为当前阶段的主要验收目标

## 当前结论

当前项目最重要的基础工作已经基本完成：

- 架构边界已经明确
- UI 已统一
- 中间层对象已固化
- 预览层和目标层已解耦
- 调试入口已经成型

后续工作的重点不该再是继续堆更多来源或更多临时入口，而是围绕明确目标：

- `Auto Rig Pro`
- `MMD`
- 映射预设
- 诊断能力

把现有框架打磨成真正可长期维护和复用的生产链路。
